# Designing a World-Class Hybrid (Neuro-Symbolic) Rule Engine: Industry & Academic Survey for Symbolica v2

## TL;DR
- **Symbolica's core bets are sound and well-supported by industry precedent** — a deterministic symbolic skeleton with explicit, accounted LLM leaves, agent-consumable traces, and a sandboxed expression language match where the entire 2023–2026 guardrails/policy-for-agents field is converging. The genuine white space Symbolica can own is *LLM-judgments-inside-declarative-rules plus replayable, machine-readable traces fed back to the agent* — no shipping engine combines both.
- **Three stated v2 positions should be revised.** (1) "Missing fact = structured error on the result" is too brittle for LLM-extracted incomplete facts — adopt FEEL-style three-valued logic or Rego-style undefined-with-explicit-default instead of CEL's hard error propagation. (2) "Fire at most once per run" is the correct *default* but needs explicit re-activation semantics (refraction keyed on fact-version, not blanket once-per-run) to avoid silent missed updates. (3) Highest-priority-wins conflict resolution is a known footgun in isolation — pair it with definition-order tiebreak (Drools' post-6.0 model) and ship overlap/shadowing static analysis.
- **Do NOT build Rete.** At 10–1,000 rules in agent loops with high fact-churn and LLM-dominated latency, Rete's incremental-match memory tradeoff does not pay off; compile rules to closures with lazy, cost-ordered predicate evaluation (cheap symbolic short-circuits before LLM leaves). Reserve a "sub-millisecond p50" claim strictly for pure-symbolic evaluation excluding any PROMPT() call, and specify ruleset size, fact count, and hardware.

## Key Findings

### Comparative Table A — Conflict Resolution & Refire Semantics (≥6 engines)

| Engine (era) | Conflict resolution | Refire / re-activation | Notes |
|---|---|---|---|
| **OPS5** (pre-LLM, 1980s) | Refraction → Recency → Specificity; two strategies LEX and MEA | Refraction: same instantiation never fires twice; can refire if facts change then re-match | The lineage source for all the below. Recency uses WME time-tags. |
| **CLIPS** (pre-LLM) | 7 strategies: depth (default), breadth, simplicity, complexity, lex, mea, random | Refraction enforced; activation removed once fired unless re-satisfied | "Difficult to say one is clearly better… without considering the specific application." Salience + strategy. |
| **Jess** (pre-LLM) | OPS5-style; salience + strategy | Refraction supported; later "slot-specific" to limit recursive looping | Per KIE, refraction is supported in Jess/OPSJ/JRules but NOT in Drools/CLIPS. |
| **Drools / Phreak** (modern) | Salience, then **rule definition order in file** (since Drools 6.0); arbitrary before | Lazy goal-oriented; agenda/activation/ruleflow groups; no-loop, lock-on-active attributes | Custom conflict resolvers removed from public API under Phreak. Docs advise "do not count on rules firing in any particular order." |
| **json-rules-engine** (modern, JS) | Priority integer per rule; stateless per run | Stateless — fires events; no truth maintenance; state lives outside engine | Facts as cached pure functions within a run. |
| **durable_rules** (modern, Py/JS/Ruby) | Rete tree (V2 fully in C, 5–10× faster, no Redis) | Facts persist; events retracted once evaluated | Synchronous assert/retract programming model. |
| **GoRules Zen / JDM** (modern) | Decision tables row-by-row, top-to-bottom, by **hit policy**; graph DAG flow | Stateless graph evaluation input→output; no agenda | Schema-first JSON; Zen Expression Language; errors halt the graph. |

**Verdict on §A:** Salience/priority survived universally but is regarded as a footgun when used for control flow; the durable industry lesson (Drools docs, CLIPS guide) is *author rules to be order-independent; use priority only to emphasize importance, not to sequence.* Refraction (don't refire the same instantiation on unchanged facts) is the conceptually correct mechanism the v1/v2 "fire at most once per run" approximates — but refraction is keyed on the *fact-set/version*, allowing legitimate refire when facts change, which blanket once-per-run does not.

### Comparative Table B — Expression Marking, Type/Missing-Data Semantics, Sandboxing

| Language | Expr vs literal | Type system & coercion | Missing-data semantics | Sandboxing / termination |
|---|---|---|---|---|
| **CEL** (Google) | Whole string is an expression; protobuf/JSON values are data | Strong static typing; **forbids type coercion** ("common source of bugs"); `T==T` only | Errors propagate; `&&`/`||` are commutative so an error/unknown on one side can be absorbed (SQL-like). Order of sub-expression evaluation unspecified. | Non-Turing-complete, linear-time, mutation-free, no side effects; "orders of magnitude faster than sandboxed JS." |
| **Rego** (OPA) | Rego rule bodies; `input`/`data` are JSON | Datalog-derived; JSON value comparison | **Undefined** propagates: missing ref → undefined → rule simply doesn't produce output; `default` keyword supplies fallbacks. Historic negation gotcha fixed via `future.keywords.not`. | Declarative, no general I/O in policy; designed for fast evaluation. |
| **FEEL** (DMN/OMG) | Boxed expressions / table cells | Rich types; `null` for invalid numbers | **Three-valued boolean logic** (true/false/null); null-friendly. | Spec-defined; Level 2 (S-FEEL) vs Level 3 conformance. |
| **Starlark** (Google/Bazel) | Python-subset scripts | Python-like; immutable data structures | Python-like name errors | **Deterministic + hermetic**: no filesystem/network/clock; safe for untrusted code; parallel-eval safe via immutability. |
| **JsonLogic** | JSON AST: `{op: [args]}`; vars via `var` | JS-ish truthiness | `var` with default arg; missing → null-ish | Tiny, no eval; data/logic separation. |
| **GoRules JDM** | Zen Expression Language in cells | Business-friendly typed expressions | Errors in Expression node halt the graph | Rust core; schema-validated (Zod schema); embeddable. |

**Verdict on §B (the most consequential finding for Symbolica):** CEL's **no-coercion** rule is the single best-validated decision in the comparison set ("CEL forbids type-coercion as this is a common source of bugs"). Adopt it. But CEL's **hard error propagation** for missing data is the *wrong* fit for "rules over LLM-extracted, often-incomplete facts." The spectrum is: CEL (error) → SQL/FEEL (three-valued, null absorbs) → Rego (undefined, rule silently doesn't fire, with explicit `default`). For an engine whose facts are routinely incomplete because an LLM couldn't extract a field, a hard error on *every* missing-fact reference (the current v2 position) will make rulesets brittle and noisy. The better design is **explicit, author-visible missing-data handling**: a distinct `MISSING`/undefined value that (a) does not silently coerce to false, (b) is surfaced in the trace, and (c) can be defaulted per-rule — closest to Rego's `default` + FEEL's three-valued logic, not CEL's error.

### §C — Hybrid Neuro-Symbolic Patterns (the differentiator)

**Production architectures combining LLM judgments with deterministic logic:**
- **NVIDIA NeMo Guardrails / Colang** (EMNLP 2023 demo, arXiv:2310.10501): a middleware layer with input/dialog/retrieval/execution/output rails. Architecture is explicitly **probabilistic**: it generates a "canonical form" of user intent via an LLM at low temperature, then uses **vector similarity** to match predefined Colang flows, optionally invoking secondary "LLM-as-judge" calls. Documented weaknesses: vector-space matching is adversarially manipulable, and programmable rails "add extra cost and latency to LLM calls, since they involve multiple steps of prompting" (NVIDIA technical-blog characterization; a Lean-4 compliance paper, arXiv:2604.01483, makes a stronger latency critique but its exact figure could not be independently verified — treat the precise number as unconfirmed). Lesson for Symbolica: keep the *skeleton deterministic and symbolic*; do not route control flow through embedding similarity.
- **Guardrails AI**: RAIL spec + Pydantic — fast/deterministic *syntactic* validation but "logically shallow"; guarantees shape, not truth.
- **Semantic routers**: fast embedding similarity to pick a tool/flow instead of a slow LLM agent call. Useful as a *cheap predicate* but non-deterministic at boundaries.
- **OPA-for-agents (2025–2026 — very active):** the dominant emerging pattern is "policy engine as intelligent proxy between agents and tools," enforcing policy at the **tool-calling layer**, gating each MCP tool invocation. Microsoft's **Agent Governance Toolkit**, open-sourced under MIT on April 2, 2026, ships an "Agent OS" — "a stateless policy engine that intercepts every agent's action before execution at sub-millisecond latency (<0.1ms p99)… Supports YAML rules, OPA Rego, and Cedar policy languages" (Microsoft Open Source Blog, Apr 2, 2026); in red-team testing it reported a 0.00% policy bypass rate versus 26.67% for prompt-based safety. It is mapped to the **OWASP Top 10 for Agentic Applications**, released by the OWASP GenAI Security Project on Dec 10, 2025 as "the culmination of input from over 100 security researchers" (ASI01–ASI10 taxonomy). OPA decision logs give built-in audit trails with correlation IDs. Oso, Permit.io, Strata Maverics, and Kuadrant MCP Gateway all converge here. **This validates Symbolica's embedded, deterministic, fast-policy positioning** — but all of these gate *around* the LLM; none run LLM judgments *inside* the rules.

**Determinism contracts for LLM calls inside rules — what practitioners actually ship:**
- **Temperature 0 is NOT deterministic.** Multiple sources (vLLM reproducibility guide, OpenAI/Gemini docs, arXiv:2503.09347) confirm batching/kernel-scheduling/floating-point non-determinism even at temp=0. arXiv:2503.09347 found GPT-4 Turbo changed 5.7% of pairwise votes across identical runs at temp 0.
- **Caching is the real determinism mechanism.** "The most reliable method is not to regenerate it at all… Normalize the prompt, model ID, and relevant parameters into a cache key… This does not make the model deterministic… but it does make your application deterministic at the interface boundary." → **This directly validates Symbolica's plan to cache on (prompt, model, temperature).** Add model *snapshot/version* to the key.
- **Seeds** only affect the sampler (help at temp>0, ignored at greedy); "best-effort," provider-documented as non-guaranteeing.
- **Self-consistency / retry-and-vote** and **multi-judge ensembles** are standard for LLM-as-judge robustness; temperature is "a core determinant of judging reliability" — low temp → near-perfect consistency, high temp → consistency collapses (arXiv:2603.28304).
- **Record/replay**: OPA's decision logs include an `nd_builtin_cache` field explicitly "intended for use in debugging and decision replay" — a concrete precedent for Symbolica's audit/replay design.

**Typed extraction — state of the art (strongly supports typed PROMPT() coercion):**
- **Constrained decoding / structured outputs** is the modern answer to bool/number coercion failures. Per OpenAI's "Introducing Structured Outputs in the API" (Aug 6, 2024): "With Structured Outputs, gpt-4o-2024-08-06 achieves 100% reliability in our evals, perfectly matching the output schemas." The model alone (trained) reached only 93% on their benchmark and gpt-4-0613 "scores less than 40%"; the 100% comes from constrained decoding via a JSON-Schema-derived **context-free grammar (CFG)**. Engines XGrammar and llguidance achieve near-zero overhead; OpenAI credited llguidance (May 2025).
- **Critical caveat repeatedly stated:** "Structured outputs guarantee shape, not truth. A model can return the right fields with the wrong values." → Symbolica must still apply semantic validation, confidence thresholds, and symbolic fallbacks after typed coercion. Also: strict mode introduces a new failure mode — per the same OpenAI announcement, "there is a new refusal string value on API responses which allows developers to programmatically detect if the model has generated a refusal instead of output matching the schema" — which must be handled as a first-class error.

**Cost/latency control patterns (validate v2 design):** budget ceilings per evaluation, **lazy evaluation order so cheap symbolic predicates short-circuit before LLM predicates** (the single highest-value pattern, aligned with CEL's commutative-operator design and json-rules-engine's fact caching), batch/parallel prompt evaluation, and caching.

**Academic neuro-symbolic work with implementable ideas:**
- **DSPy Assertions / LM Assertions** (Singhvi et al., arXiv:2312.13382, Dec 2023): `dspy.Assert` (hard — halts pipeline after `max_backtracking_attempts` with AssertionError) vs `dspy.Suggest` (soft — logs failure, continues "best-effort"). On failure, the framework **backtracks and retries the failing module, injecting the erring output + error message into the retry prompt** for self-refinement. Verbatim: "LM Assertions improve not only compliance with imposed rules but also downstream task performance, passing constraints up to 164% more often and generating up to 37% more higher-quality responses." → **Directly implementable for Symbolica:** a PROMPT() that fails typed coercion or a confidence threshold should support assert (error) vs suggest (retry-with-feedback, then fall back symbolically) semantics.
- **Scallop** (PLDI 2023, arXiv:2304.04812): Datalog + provenance semirings for discrete/probabilistic/**differentiable** reasoning, with Python/PyTorch bindings. Relevant idea: relational/Datalog representation and provenance tracking for *why* a fact holds — useful conceptually for Symbolica's traces, though full differentiable reasoning is out of scope for a deterministic engine.

### §D — Explainability & Decision Auditing

- **DMN Decision Requirements Graph (DRG/DRD)** (OMG spec, namespace 20191111): a self-contained dependency graph tracing a decision through sub-decisions, input data, and knowledge sources, serialized in XML under `<definitions>`. This is the canonical model for *decision decomposition* traces and validates Symbolica's DAG-ordered, dependency-aware design.
- **The closest thing to a machine-readable decision-trace standard is OPA's decision log schema.** Exact fields include: `decision_id` (per-decision traceability), `trace_id`/`span_id` (**W3C trace-context compliant**), `path`, `input`, `result`, `bundles[_].revision`, `timestamp` (RFC3339), `metrics`, `erased`/`masked` (JSON Pointers to redacted fields), `nd_builtin_cache` (for **decision replay**), and **`ids` = annotation IDs of rules that were successfully evaluated (which rules fired)**. → Symbolica should emit a trace that is essentially a superset of this: fired rules + IDs, field values consumed, intermediate steps, PROMPT() call records (prompt, model, response, cost, latency, cache hit), W3C-trace-context IDs, and a replay cache.
- **W3C PROV (PROV-DM / PROV-O)** is the W3C-recommendation provenance standard ("information about entities, activities and people involved in producing a piece of data") used in decision-provenance/accountability research (arXiv:1804.05741; explainability-by-design arXiv:2206.06251, implemented with the UK ICO on a loan-decision scenario). Emerging **PROV-AGENT** (2025, arXiv:2603.21692) extends PROV for AI-agent workflows but is not yet widely adopted.
- **"Why-not" / regulatory mandate for specific reasons:** US **ECOA / Regulation B, 12 CFR §1002.9(b)(2)** requires adverse-action reasons to be "specific" — explicitly: "Statements that the adverse action was based on the creditor's internal standards or policies or that the applicant… failed to achieve a qualifying score… are insufficient." CFPB interpretation: "disclosure of more than four reasons is not likely to be helpful to the applicant." CFPB has confirmed this applies regardless of black-box/AI model complexity. **FICO reason codes** are the industry mechanism — ranked score factors delivered with a score, "may be used… when taking adverse action," listed most-influential first, up to 4 (a 5th for inquiries). → Symbolica's near-miss/"why-not" output should mirror this: ranked failed conditions with the specific field and threshold, capped at a small number.
- **Counterfactual / contrastive explanations (the "what would flip it" literature):** Wachter, Mittelstadt & Russell (Harvard JOLT 2018, arXiv:1711.00399) define a counterfactual explanation as "the smallest change to the world… to obtain a desirable outcome… without needing to explain the internal logic of the system." Human explanations are "intrinsically contrastive" ("Why P rather than Q?"); a counterfactual specifies "minimal conditions for a given data instance to be classified differently." Dhurandhar et al. (NeurIPS 2018) "pertinent negatives"; VLDB (PVLDB Vol. 16) "Computing Rule-Based Explanations by Leveraging Counterfactuals" is directly applicable to rule engines.
- **Does feeding traces back to an LLM help? (empirical, 2023–2026):** Yes, conditionally. LLM-generated rationales "can significantly enhance downstream model performance" (arXiv:2508.09776); **rationale order matters** — "performance… strongly depends on whether the rationale is output before or after the answer" (arXiv:2505.00776); **structured/compact** input "naturally leads to shorter reasoning traces and substantial token savings" (Focused-CoT, arXiv:2511.22176); but there is overthinking/diminishing-returns risk from overly long traces, especially for smaller models (arXiv:2508.18743). → Symbolica should emit traces in a **structured, compact** form (not verbose prose), placed *before* the agent's next decision, and make verbosity configurable.

### §E — Temporal & Streaming Reasoning
- CEP windowing primitives (Flink docs): **tumbling** (non-overlapping), **sliding** (fixed-size overlapping, defined by size + slide), **session** (gap-based), **global**. **Event time vs processing time**: event time "provides deterministic results and handles out-of-order events correctly." **Watermarks** = an assertion that "all events with timestamps less than T have arrived," used to trigger window computation and bound lateness; **allowed lateness** lets late events still be included.
- **Minimal subset for an embedded, non-distributed engine:** event-time timestamps on facts; sliding/tumbling windowed aggregates; a single bounded-lateness/"grace" parameter instead of full watermark machinery. Kai Waehner's CEP guidance: if a pattern is "expressible as a windowed aggregation rather than a strict event sequence, a standard windowed query is simpler" than CEP — so Symbolica does NOT need a CEP state machine for `recent_avg`/`sustained_above`.
- **`sustained(condition, duration)` done correctly:** the v1 80%-coverage heuristic is arbitrary and non-principled. The defensible definition borrows from event-time semantics: *the condition holds at every observed sample within `[now − duration, now]`, evaluated over event-time timestamps, with an explicit policy for gaps* (either "require continuous coverage with max-gap G" or "require the condition true across the full window with no observed false sample"). Make the gap/coverage policy an explicit, documented parameter — not a magic 80%.

### §F — Verification, Testing & Governance
- **DMN decision-table analysis is the mature, practical model at 10–1,000 rules.** Hit policies (Unique/Any/Priority/First/Collect) plus automated checks for **completeness (gaps)**, **overlap**, and **subsumption/shadowing**. Tools (Trisotech, Sparx EA, Camunda) detect "gaps in the rules," "overlap" (a violation under Unique), and subsumption. Calvanese et al. (arXiv:1603.07466) formalize DMN-table semantics and analysis (geometric algorithms for overlap/gap detection). → Symbolica should ship static analysis: unreachable rules, contradictory verdicts, shadowed rules, and (where verdicts are categorical) completeness over declared input domains.
- **CI for rules:** golden/conformance suites (CEL ships a conformance test suite), property-based testing over fact spaces, shadow/champion-challenger execution, ruleset versioning + rollback. The Drools-6 lesson — *load order preserved across versions* — matters for reproducibility under change.

### §G — Performance & Benchmarking
- **Rete's tradeoff** (Forgy 1974/1979/1982; corroborated by Oracle's NRE docs): Rete "trades increased memory use for improved performance," predicated on "facts… change slowly over time through a series of inference cycles." Oracle explicitly notes "many business rules use cases do not fit this usage profile and thus do not benefit from… Rete while incurring the… memory consumption." Drools moved to **Phreak** (lazy, goal-oriented) precisely because eager Rete wastes work.
- **Conclusion: do not build Rete.** Symbolica's profile — small rulesets, facts rebuilt per agent turn (high churn, low inter-run persistence), once-per-run firing, latency dominated by LLM calls — is the *opposite* of Rete's sweet spot. Compile rules to **closures** with lazy, cost-ordered predicate evaluation. Rete pays off only at large rule×fact products with high fact persistence and small per-cycle deltas.
- **Credible "sub-millisecond" claim must specify:** pure-symbolic only (no PROMPT()), ruleset size (e.g., p50 at N=100 rules), fact count, warm vs cold, hardware/runtime, and percentile (p50 vs p99). CEL/OPA benchmark suites and the OPS5 manners/waltz benchmarks are the lineage references.

### §H — Product & Ecosystem Lessons
- **Why Drools-class engines were displaced:** heavyweight JVM embedding, steep DSL, Rete memory overhead for non-fitting workloads, and operational complexity. Lightweight winners (OPA/Rego, CEL, GoRules Zen, json-rules-engine) won on **embeddability, fast non-Turing-complete evaluation, schema-first governance, and learnability**.
- **White space for Symbolica:** the 2024–2026 "rules/policy for agents" field (OPA-for-agents, Microsoft Agent Governance Toolkit, Oso, Permit.io, NeMo Guardrails) universally gates *around* the model and treats the LLM as an opaque external call. **No shipping engine offers (a) LLM judgments as first-class, typed, cost-accounted predicates *inside* declarative rules, plus (b) replayable, structured, agent-consumable reasoning traces.** That intersection is Symbolica's defensible niche.

## Design Recommendations for Symbolica v2

1. **Keep no-coercion equality/typing (adopt CEL's rule).** Evidence: CEL spec — coercion is "a common source of bugs." Confidence: **High**. **Confirms** v2's typed return coercion *only if* coercion is limited to explicit PROMPT() return typing, not silent cross-type comparison.

2. **Replace "missing fact = hard error" with explicit undefined/three-valued semantics + per-rule defaults.** Evidence: Rego undefined-propagation + `default`; FEEL three-valued logic; the fact that Symbolica's facts are LLM-extracted and routinely incomplete. Confidence: **High**. **CONTRADICTS** the stated v2 position ("referencing a missing fact is a structured error"). A hard error on every missing reference will make real rulesets brittle; instead surface a distinct `MISSING` value in the trace, never coerce it to false, and let authors default it. *Keep* hard errors for type/parse violations.

3. **Keep "fire at most once per run" as default, but key it on fact-version (refraction), with explicit re-activation.** Evidence: OPS5/CLIPS refraction; risk of silently missing legitimate re-fires when facts change mid-run. Confidence: **Medium-High**. **Partially contradicts** the blanket once-per-run position: implement refraction (don't refire the same rule on the same fact-set) rather than once-per-run-period, so a rule can legitimately re-fire if its inputs change in a later pass.

4. **Conflict resolution: highest-priority-wins, tiebroken by definition order, plus static overlap analysis.** Evidence: Drools 6.0 model (salience then file order); universal "don't sequence with salience" warning. Confidence: **High**. **Refines** the v2 position — priority alone is a footgun; add deterministic definition-order tiebreak and ship conflict/shadow detection so authors *see* when two rules produce contradictory verdicts.

5. **Compile to closures with lazy, cost-ordered predicate evaluation; do NOT build Rete.** Evidence: Oracle NRE docs, Phreak's lazy redesign, Symbolica's high-churn/small-ruleset/LLM-latency profile. Confidence: **High**. **Confirms** v2's pass-based forward chaining. Order predicate evaluation cheapest-symbolic-first so PROMPT() leaves are only reached when they can change the outcome.

6. **Determinism contract for PROMPT(): cache on (prompt, model, model-snapshot, temperature, schema); record/replay; never claim temp-0 determinism.** Evidence: vLLM/OpenAI docs; arXiv:2503.09347; OPA `nd_builtin_cache` replay precedent. Confidence: **High**. **Confirms and strengthens** v2's caching plan — add model snapshot to the key and a record/replay store for audits/tests.

7. **Use constrained decoding / JSON-schema structured outputs for typed PROMPT() returns; validate truth separately; handle refusals.** Evidence: OpenAI structured outputs (CFG, 100% reliability of shape on gpt-4o-2024-08-06); "shape not truth" caveat; refusal `refusal` string failure mode. Confidence: **High**. **Confirms** typed coercion but adds: shape guarantee ≠ correctness, so retain confidence thresholds + symbolic fallback.

8. **Adopt DSPy-style assert vs suggest semantics for failed LLM predicates.** Evidence: arXiv:2312.13382 (backtrack-and-retry with error feedback; up to 164% better constraint satisfaction, 37% higher-quality responses). Confidence: **Medium-High**. **Extends** v2: a PROMPT() that fails coercion/confidence can retry-with-feedback N times (suggest) then fall back to a symbolic default, or hard-error (assert).

9. **Emit a structured, compact, replayable trace modeled on OPA decision logs + W3C trace-context.** Evidence: OPA schema (`ids`, `decision_id`, `trace_id`, `nd_builtin_cache`); empirical results that structured, rationale-before-answer traces help downstream LLMs. Confidence: **High**. **Confirms** v2's explainability-first design; specify the schema explicitly and keep verbosity configurable.

10. **Ship "why-not"/near-miss output as ranked failed conditions (field + threshold), capped small.** Evidence: ECOA §1002.9(b)(2) "specific reasons" + CFPB "more than four… not likely to be helpful"; FICO reason codes (ranked, ≤4); Wachter counterfactuals; VLDB rule-based counterfactuals. Confidence: **High**. **New capability** — for each non-firing rule of interest, report which condition failed and the value that would have flipped it.

11. **Temporal: implement event-time windowed aggregates with one explicit bounded-lateness parameter; redefine `sustained()` rigorously.** Evidence: Flink event-time/watermark semantics; Kai Waehner's "windowed aggregation, not CEP" guidance. Confidence: **High**. **Refines** v1 — replace the arbitrary 80% heuristic with an explicit coverage/max-gap policy over event-time samples.

12. **YAML: parse with explicit-typing / quote-strings discipline; validate against a schema at compile time; consider a schema-first carrier.** Evidence: the Norway problem (`NO`→false), version-number/leading-zero coercion; StrictYAML/Kyaml; GoRules' Zod-schema JDM. Confidence: **High**. **Confirms** v2's compile-time validation; additionally disable YAML 1.1 implicit booleans and require quoting or explicit tags. Symbolica's `"= expr"` sigil for expressions (everything unmarked is a literal) is a *good* mitigation — it sidesteps the "is this string code or data" ambiguity that bites JsonLogic/CEL embeddings.

13. **Ship DMN-style static analysis at build time: unreachable, contradictory, shadowed, incomplete.** Evidence: DMN completeness/overlap/subsumption checks; Calvanese et al. formalization. Confidence: **High**. **New governance capability** matching the immutable-after-build model.

14. **Keep the engine immutable after build and do all validation at compile time.** Evidence: Starlark hermetic/deterministic model; CEL compile-time type-checking; reproducibility needs. Confidence: **High**. **Confirms** v2 positions.

### Staged rollout & thresholds that would change these recommendations
- **Stage 1 (must-ship for v2):** no-coercion typing (#1), undefined+default missing-data semantics (#2), closure compilation with cost-ordered laziness (#5), PROMPT() caching + record/replay (#6), structured outputs + refusal handling (#7), structured replayable trace (#9), YAML hardening (#12). These are low-risk and well-evidenced.
- **Stage 2 (fast follow):** fact-version refraction (#3), definition-order tiebreak + overlap analysis (#4), DSPy assert/suggest for PROMPT() (#8), near-miss "why-not" output (#10), rigorous `sustained()` (#11).
- **Stage 3 (governance maturity):** full DMN-style static analysis suite (#13), conformance/golden-suite CI, champion-challenger shadow execution.
- **Thresholds that would reverse "don't build Rete":** if real workloads show (a) facts persisting across many runs with small per-run deltas, (b) rulesets routinely >5,000 rules, AND (c) pure-symbolic evaluation becoming the latency bottleneck (not LLM calls), then incremental matching becomes worth reconsidering. Until all three hold, closures win.
- **Threshold that would change the missing-data recommendation:** if rulesets are authored over *validated, schema-complete* facts (not LLM-extracted), CEL-style hard errors become defensible. The recommendation is conditioned on incomplete LLM-extracted facts.

## Open Debates (no industry consensus)
- **Missing-data philosophy:** error (CEL) vs undefined (Rego) vs three-valued null (FEEL/SQL) vs Peterfreund–Libkin 2-valued "null→false" (arXiv:2307.15751). All have adherents; Symbolica must pick deliberately (recommendation: undefined + explicit default + trace visibility).
- **Optimal trace length/shape for downstream LLMs:** concise-structured (Yao et al.) vs longer-reasoning scaling laws (Snell et al., Shen et al.) vs overthinking degradation — unresolved; make it configurable.
- **Whether seeds/temperature-0 give any useful determinism:** providers say best-effort only; community split on whether to bother vs cache-only.
- **CEP vs windowed-aggregation for "sustained" patterns:** whether sequence/order ever matters enough to justify a state machine in an embedded engine (likely no for Symbolica).
- **Salience's very existence:** Drools maintainers have stated intent to "completely remove the need for salience" — open question whether priority should exist at all vs pure dependency ordering.

## Caveats
- **Recency/source quality:** §C and §H rely on 2024–2026 sources; several agent-policy claims (Microsoft Agent Governance Toolkit, OWASP Agentic Top 10) come from vendor/blog announcements rather than peer review — the *direction* (policy-engine-as-proxy, sub-ms enforcement) is corroborated across multiple independent sources, but specific performance numbers are self-reported.
- **One unverified figure:** the "hundreds of milliseconds" latency cost attributed to NeMo's secondary LLM-judge calls could not be independently confirmed against the cited Lean-4 paper; the weaker, well-sourced claim (rails "add extra cost and latency… multiple steps of prompting") stands.
- **DMN spec citations** were corroborated via vendor docs (Red Hat, Drools, Camunda) that quote the OMG specification directly rather than the OMG PDF itself.
- **LLM-trace-helps-downstream evidence is task-dependent:** rationales help complex reasoning more than simple NLU, and order/length matter — so the "feed traces back" recommendation should be validated on Symbolica's specific agent workloads, not assumed universally beneficial.
- **§D's "machine-readable decision-trace standard"** is an emerging de-facto convergence (OPA decision logs + W3C trace-context + W3C PROV), not a single ratified standard for *rule-engine* traces specifically; Symbolica would be defining schema in a still-forming space.
- **Pre-LLM-era sources** (OPS5, CLIPS, Jess, Forgy's Rete) are explicitly marked; their conflict-resolution and matching lessons remain valid but predate agent/LLM workloads, so their conclusions are applied by analogy, not direct transfer.
